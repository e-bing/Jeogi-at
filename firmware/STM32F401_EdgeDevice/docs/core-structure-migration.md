# Core Structure Migration

## Purpose

This project now organizes user code inside `Core` by responsibility while keeping CubeMX-generated files easy to identify.

The change had four goals:

- separate generated board code from user logic
- separate hardware-facing drivers from higher-level services
- separate product features from shared application state
- normalize user file names to lowercase `snake_case`

## Ownership Rules

- `app`: top-level app orchestration
- `common`: shared runtime state and common data
- `drivers`: direct HAL or device-facing code
- `services`: reusable functional logic built on top of drivers
- `features`: product-facing behavior such as UART commands, display flow, and sensor feature logic

## Include Convention

User headers are now included from the `Core/Inc` root using ownership-based paths.

Examples:

- `#include "app/app_task.h"`
- `#include "common/data_manager.h"`
- `#include "drivers/display/gui_paint.h"`
- `#include "features/protocol/uart_handler.h"`
- `#include "services/audio/audio_player.h"`

## Naming Convention

- user file names: lowercase `snake_case`
- generated STM32 files: keep CubeMX naming
- directories: lowercase
- include paths should show ownership

## Notes

- This change focused on structure and naming. Public function names were largely kept as-is to reduce behavioral risk.
- `CMakeLists.txt` was updated to reflect the new file layout and include paths.
